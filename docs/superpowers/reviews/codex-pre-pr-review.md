# Codex Pre-PR Review ‚Äî Routine Detail Page

**Note:** Codex CLI was not available in the local environment when this review ran. This is a deep pass with Opus standing in. The 2026-04-27 00:58 UTC scheduled agent will run an additional Codex review remotely.

**Reviewed:** commits `38de1f5f..5eefe658` on `feat/routine-detail-page` (`apps/client/` only).

## Summary

The feature is structurally well organised: route decomposition is clean (`/routines`, `/routines/$routineId`, `/routines/$routineId/runs/$executionId`), the sidebar is properly URL-driven, and the per-task review-fix commits show good iterative discipline. Tests are broad (76 component + 20 route tests) and the deletion of `RoutineSettingsDialog`/`RoutineSettingsTab` is complete with no dangling imports.

The blocking concerns are at the routing layer ‚Äî specifically, the new `$routineId.tsx` parent route does not render `<Outlet />`, so under TanStack Router v1.141 the child run-route component will never mount at runtime even though every direct-component test passes. There are also several Important items: stale-data ordering on Last-Run, an auto-expand effect that resurrects the user's manual sidebar collapse on remount, four newly added i18n keys that ship only in English, the unresolved `<div role="button">` interactive nesting in `RoutineCard`, and STATUS_COLORS duplicated five times across the routines surface. None of the Important items individually blocks merging, but they are worth fixing in this PR rather than as follow-ups.

## Strengths

- Routes cleanly split: index renders sidebar+empty state, detail renders sidebar+`RoutineDetailView`, run-page renders sidebar+`ChatArea`+`RightPanel`. Each owns its layout cleanly.
- `RunListItem` is a focused, well-tested abstraction shared by Overview and Runs tabs; the trigger-badge mapping (Retry first, then triggerType) matches the spec.
- Index route owns the picker and `CreateRoutineDialog` and passes `onRequestCreate` to the sidebar so both the empty-state CTA and the sidebar `+` open the same picker ‚Äî single source of truth (good architectural call).
- Review-fix commits are visible and meaningful: keyboard double-fire fix on inner buttons (`stopButtonKeyDownPropagation`), chevron-icon fix that keeps `ChevronRight` when there's no expandable content, run-not-found fallback with a real `<Link>`, and the `retry: 0` on the detail-route `getById` so a deleted/forbidden routine doesn't loop forever.
- Test discipline: route-level tests cover the loading, error, not-found, and creation-sentinel states, the validateSearch contract is tested both for accepted values and for stripping unknown values, and `RunListItem` covers the full trigger badge matrix including `retry`.

## Findings

### Critical

- `apps/client/src/routes/_authenticated/routines/$routineId.tsx:30` (and `‚Ä¶/$routineId.runs.$executionId.tsx:16`) ‚Äî the parent route `$routineId.tsx` defines a `component` (`RoutineDetailPage`) and never renders `<Outlet />`, but the child route `$routineId.runs.$executionId.tsx` is parented under it (see `routeTree.gen.ts:265` ‚Äî `getParentRoute: () => AuthenticatedRoutinesRoutineIdRoute`, and the `AuthenticatedRoutinesRoutineIdRouteWithChildren` declaration). TanStack Router v1.141 only mounts a child match through `<Outlet />` (see `node_modules/@tanstack/react-router/dist/esm/Match.js:144-150` and `Match.js:218-236` ‚Äî `if (!childMatchId) return null;`). At runtime, navigating to `/routines/$id/runs/$execId` will render `RoutineDetailPage` (parent) and the child `RoutineRunPage` will never mount, so `ChatArea` + `RightPanel` will not appear on that URL. The existing tests bypass this by invoking `Route.__config.component` directly, so the bug is invisible to the test suite. Fix one of two ways: (a) make `$routineId.tsx` a layout ‚Äî render `<Outlet />` only and move the detail content into a new `$routineId.index.tsx`; or (b) keep the current shape but in `RoutineDetailPage` detect the child match (e.g., `useMatches` or `useChildMatches`) and render `<Outlet />` instead of the detail view when a child is active. Add an integration test that exercises both URLs through an actual `RouterProvider` so this can't regress. Same pattern exists in the wiki feature today and may be a latent bug there too ‚Äî worth noting separately, but it does not change the verdict for this branch.

### Important

- `apps/client/src/components/routines/tabs/RoutineOverviewTab.tsx:77-80` ‚Äî `lastRunAt = routine.currentExecution?.execution.startedAt ?? executions[0]?.startedAt ?? null` does not pick the most recent of the two as the spec requires (spec ¬ß6, step 2: "whichever is most recent"). Because `routine` and `executions` are independent queries with independent poll cadences, the `currentExecution` payload can be older than `executions[0]` mid-flight ‚Äî the user sees an out-of-date "Last run". Fix by comparing both timestamps and using `Math.max(currentStartedAt, latestStartedAt)`.
- `apps/client/src/components/routines/RoutinesSidebar.tsx:62-70` ‚Äî the `useEffect` that auto-expands `selectedRoutineId` does guard against re-adding within the same mount (`if (prev.has(selectedRoutineId)) return prev;`), but `expandedRoutineIds` is local state. Navigating from `/routines/$id` to `/routines/$id/runs/$execId` (and back) re-mounts the sidebar (each parent route has its own component tree under the same `_authenticated` Outlet), so the user's manual collapse via chevron is silently lost on every navigation. Lift the expansion set into a Zustand store or a React Query placeholder query, or persist in `sessionStorage`. (This is the "auto-expand re-firing after manual chevron collapse on remount" item the brief flagged.)
- `apps/client/src/components/routines/RoutineCard.tsx:160-166` ‚Äî the card header is still a `<div role="button" tabIndex={0}>` containing real nested `<button>` children (chevron, Play, Settings). This is invalid interactive nesting per WAI-ARIA / HTML spec; screen readers may announce the inner buttons as part of the outer button's accessible name. Task 6 added `stopPropagation` for keyboard activation, which is the right escape valve, but the structural problem remains. Fix by giving the title-row its own real `<button>` (or `<Link>`) and keeping the chevron / Start / Settings as sibling buttons outside it. (This is a refactor that touches `RoutineCard.test.tsx`'s chevron query selector ‚Äî keep the rename in scope.)
- i18n parity for four new keys: `historyTab.running` (used at `RunListItem.tsx:85`), `detail.more` (used at `RoutineDetailView.tsx:105`), `detail.cancel` (used at `RoutineDetailView.tsx:183`), and `detail.notFound` (used at `$routineId.tsx:80`) live only in `apps/client/src/i18n/locales/en/routines.json` (or only as inline fallbacks in code). Verified by parsing all 12 locale JSON files. The runtime degrades gracefully because every call site provides an English fallback, but non-English users see the English string interleaved with the localized UI. Backfill these four keys across `de`, `es`, `fr`, `it`, `ja`, `ko`, `nl`, `pt`, `ru`, `zh-CN`, `zh-TW`. The other nine new `detail.*` keys (`createdAt`, `lastRunAt`, `totalTokens`, `currentRun`, `view`, `recentRuns`, `viewAllRuns`, `showMore`, `runNotFound`, `backToRoutine`, `toggleExpand`) are properly localized in all 12 locales ‚Äî confirmed.
- `STATUS_COLORS` is duplicated five times across the routines surface ‚Äî `RoutineDetailView.tsx:33`, `RoutineCard.tsx:19`, `RunItem.tsx:10`, `RunListItem.tsx:19`, and `tabs/RoutineOverviewTab.tsx:21`. All five copies are identical right now (verified), but the moment one needs to change (e.g. status added, color tweaked) drift is inevitable. Extract to `@/lib/routine-status.ts` (or similar) and import. Low risk, mechanical change.

### Minor / observations / follow-up

- `apps/client/src/routes/_authenticated/routines/$routineId.tsx:43` ‚Äî `refetchInterval: (query) => (query.state.error ? false : 5000)` polls every 5 s for the entire time the detail page is mounted, regardless of which tab is active. Triggers / Documents tabs do not surface live-run state, so the poll is wasted there. Consider gating the interval on `tab !== "documents" && tab !== "triggers"` or pausing polling when the page is hidden (`document.visibilityState`).
- `apps/client/src/routes/_authenticated/routines/$routineId.runs.$executionId.tsx:24-27` ‚Äî the conditional refetch interval (`30000` if `taskcastTaskId`, else `5000`) is the right idea but `query.state.data?.currentExecution?.execution.taskcastTaskId` reads the previously-fetched value, so the first refetch always uses the seed cadence. Acceptable; documenting because the comment in the file does not call this out.
- `apps/client/src/components/routines/RunListItem.tsx:85` ‚Äî `t("historyTab.running", "running")` returns the English fallback in non-English locales (see Important note above). The interpolation is `running 5m 0s+`, which won't grammatically work in many locales (e.g., Russian / Japanese word order). When backfilling, consider switching to a single key with `{{duration}}` interpolation rather than concatenating two strings.
- `apps/client/src/components/routines/tabs/RoutineRunsTab.tsx:26` ‚Äî `useState(PAGE_SIZE)` resets to 20 each time the Runs tab becomes active because Radix unmounts inactive `<TabsContent>` (the inline comment correctly notes this is intentional). Acceptable trade-off, but worth confirming the spec wanted reset-on-switch rather than reset-on-route-change.
- `apps/client/src/routes/_authenticated/routines/$routineId.runs.$executionId.tsx:90-96` ‚Äî the run-not-found fallback uses a real `<Link>` (good), but the surrounding container has no landmark role and no auto-focus. A user landing here via a stale URL gets a screen-reader-silent panel. Add `role="status"` or `role="region" aria-labelledby={...}` and consider auto-focusing the back link. Minor a11y polish.
- `apps/client/src/components/routines/RoutineDetailView.tsx:175-179` ‚Äî the AlertDialog title key is `settingsTab.deleteTitle` (with English fallback), but `settingsTab.deleteTitle` is not defined in any locale file. Same gentle issue as the four English-only keys. Consider moving to `detail.deleteTitle` for consistency with the rest of the deletion flow (`detail.delete`, `detail.deleteConfirm`).
- `apps/client/src/routes/_authenticated/routines/__tests__/$routineId.runs.$executionId.test.tsx:121-302` ‚Äî the test suite covers unknown-execution ‚Üí not-found and the creation sentinel happy path. It does not cover (a) the routine query rejecting/404'ing while on `/runs/$execId` (currently the test at `:122` only exercises the loading state via a never-resolving promise), and (b) `/runs/creation` for a routine whose status transitions from `draft` to non-draft mid-view. Adding both as cases would have caught the related Important issues had they been bugs.
- `apps/client/src/components/routines/__tests__/RoutinesSidebar.test.tsx:215-219, 403-413` ‚Äî the chevron selector relies on the test's `t()` mock returning the bare key (`'detail.toggleExpand'`) when no fallback is supplied. The production code at `RoutineCard.tsx:174` calls `t("detail.toggleExpand")` (no fallback). If anyone adds a fallback string to that call, every chevron test breaks silently. Consider switching the selector to `aria-label` matching with a regex, or adding a stable `data-testid` on the chevron button.
- `apps/client/src/components/routines/__tests__/RoutineDetailView.test.tsx:64-139` ‚Äî the inline mock of `@/components/ui/dropdown-menu` substitutes a hand-rolled context provider for Radix's primitives. This is necessary in jsdom, but the mock skips `aria-haspopup`, `aria-expanded`, focus management, and the asChild wiring. Tests pass but they do not exercise the real interactive surface. Acceptable for unit scope; add a Playwright e2e for the kebab ‚Üí Delete ‚Üí confirm flow as part of the Routine detail page e2e suite.
- `apps/client/src/components/routines/RunListItem.tsx:150-153` ‚Äî `t("detail.tokenCount", { count, defaultValue: \`${count} tokens\` })`mixes i18next interpolation with a templated default. When pluralization rules eventually land for non-English locales, this`defaultValue` will paper over a missing plural form rather than surfacing it. Low priority; just note the trade-off.
- The `executionId === "creation"` sentinel is a load-bearing string. The collision argument in the brief is theoretical ‚Äî UUIDv4 cannot equal `"creation"` ‚Äî but the sentinel only runs through the run-route file. There is no central constant for it. Extract `const CREATION_SENTINEL = "creation"` somewhere shared (`lib/routines.ts`?) so search and rename are easy. Minor.
- Validate-search at `$routineId.tsx:19-27` correctly strips unknown tab values to `{}` (test at `__tests__/$routineId.test.tsx:184-199` confirms). No issue, just noted.
- Dead code / unused-imports check passed: `RoutineSettingsDialog`, `RoutineSettingsTab`, `RoutineList` are deleted and no production file imports them. `services/api/routines.ts` still exports `RoutineListParams` ‚Äî that is the API list-query DTO, not the deleted component, so leave it.

## Assessment

PR-ready after the **Critical** Outlet-missing fix is verified ‚Äî that one is potentially shipping a non-functional run-page route, and the suite cannot catch it because every test invokes `Route.__config.component` directly.

After that, I would also fix the four Important items in this PR rather than as follow-ups: the `lastRunAt` ordering bug is a one-line spec deviation, the auto-expand-on-remount issue is a UX regression vs. the legacy modal flow, the four-key i18n parity gap is mechanical, and the `RoutineCard` `<div role="button">` is the kind of a11y debt that compounds. STATUS_COLORS extraction is also small enough to roll in.

Everything in the Minor / observations list is fine as follow-up.
